# テスト規律 — 振る舞いに対してテストを書く

fed の TDD ワークフロー（tdd-v5 / tdd-v6 等）の **全 agent が共通で従う原則** を定義する正本。テストを書く側（test-implementer / implementer / fixer）と、テストを論じる側（code-reviewer-* / code-review-integrator）の双方からここを参照する。

「テストは振る舞いに対してしか書かない」は決め打ちチェックリストではない。リストに無いアンチパターンは無数にある。**判定基準そのもの** を持って、新しいケースに自分で当てて判別できるようにすること。これが本ファイルの目的。

## 原則

テストは **振る舞い（観測可能な契約）** に対して書く。実装の「形」や差分の「写し」に対しては書かない。

振る舞いの典型: 公開 API のレスポンス、永続化されたデータの状態、送出された例外の種別、呼び出し元から見える代替経路、外部スキーマ、セキュリティ境界。

差分の写しの典型: 内部関数の名前・呼び出し順、private な状態、組み立てられた SQL 文字列・ログ文字列、生成された内部 ID、削除されたシンボルの不在そのもの。

## 判定基準

新しいテスト（または「テストを足せ」というレビュー指摘）に出会ったとき、必ず次の **2 つの基準** を当てて判別する。リストに無いアンチパターンも、この基準で機械的に裁ける。

### 基準 1: 置換不変性

> そのテストは、**同じ振る舞いを実現する別アプローチの実装** に書き換えても、通り続けるか？

通り続ける → 振る舞いベース。実装の自由度を奪わない。

通らない → 差分追従。実装を別の同等な方法で書いた瞬間に壊れるテストは、実装の「形」に張り付いている。書いてはならない。

### 基準 2: 観測可能性

> そのテストが固定している事実は、**システムの外側から観測できる接点** に対応しているか？

外側から観測できる接点（観測可能）:
- 公開 API のリクエスト／レスポンス
- 永続化されたデータの形（DB レコード、ファイル、キャッシュ等）
- 送出された例外の種別（catch 側から見える）
- 呼び出し元の経路（どのメソッドが返るか）
- 外部スキーマ・契約（JSON Schema、protobuf、OpenAPI、設定ファイル形式）
- セキュリティ境界（認可結果、サニタイズ後の値）

内側だけで起こること（観測不可）:
- private 状態、内部の中間変数
- 内部ヘルパー関数の呼び出しトレース
- 組み立てたクエリ文字列・ログ文字列の中身
- 内部生成された UUID / 連番 ID の値そのもの
- コードのトポロジー（どのファイルにある、どのクラスから呼ばれている）

両方の基準を **AND** で要求する。「置換不変だが観測不可」「観測可能だが置換不変でない」のどちらも差分追従の疑いがある。

## レビュー指摘を受け取ったときの翻訳手順

レビュー指摘で **最も多く差分追従テストが生まれるのはレビュー対応サイクル**。指摘の文面通りにテストを書くと、レビュアーが意図せず差分の形を提示してしまっていた場合、そのまま実装してしまう。

そこで、テスト追加・修正系の指摘を受け取った側（implementer / fixer / 人間開発者）は、**必ず次のフィルタを通す**。指摘の出どころが agent でも人間でも同じ扱い。

1. **その指摘は「現在こう動く」を固定しようとしているか、「以前と違って今はこう」を固定しようとしているか** を見分ける（後述「不在を assert したいと感じたとき」の判別の問い）
   - 後者なら、たいていテスト不要。git diff・スキーマファイル等が既に担保しているはず
   - 前者なら、観測可能な振る舞いに翻訳できるか考える:
     - SQL クエリの組み立て方 → そのクエリが返す結果に対する公開 API の挙動
     - 内部関数の呼び出し有無 → その呼び出しが生む外部から観測される副作用
     - 削除されたメソッドが担っていた要件 → 新しい設計で同じ要件がどう実現されるか（削除を縛るのではなく新契約をテスト）
     - 機密フィールドを返さない → レスポンスのフィールド集合を allowlist で完全一致
2. **翻訳できる → その振る舞いをアサートするテストを書く**。指摘の文面通りに `not_called` / `toBeUndefined` / 文字列 equal を書かない
3. **翻訳できない or 動機が差分起点 → 対応見送り**。レビュー対応サマリーに「テストすべき観測可能な振る舞いが無いため対応見送り（git diff が担保）」のように一行記録する。これは「テストは振る舞いに対してしか書かない」原則に基づく **正当な却下** であって、サボりではない

人間レビューでも同じ手順を踏む。「`X` が削除されたなら、`X` が存在しないことを assert すべき」と言われても、ほぼ全ての場合テスト不要（後述「不在を assert したいと感じたとき」を参照）。allowlist 完全一致や呼び出し側の振る舞いとして翻訳できる場合のみテストを書き、それ以外は対応見送りを人間に返す。

## 典型例（非網羅）

判定基準を当てた結果として **頻出する** 違反パターン。**このリストは網羅ではない**。リストに無いものでも、判定基準で差分追従と判定したら同じ扱い。

### 違反: 削除されたシンボルの不在チェック
```js
expect(oldHelper).toBeUndefined()
```
```python
assert not hasattr(obj, "old_method")
with pytest.raises(AttributeError):
    obj.old_method()
```
削除は `git diff` が表す事実。テストで重ねても情報は増えず、誰かが再追加した時にしか壊れない。

**書くべきもの**: 削除に伴って観測される振る舞いの変化（呼び出し元が別経路を通る／公開 API のレスポンスが変わる／代替エラーが送出される）。

### 違反: 内部関数の呼び出し有無を spy で固定
```js
expect(internalHelper).not.toHaveBeenCalled()
expect(spy).toHaveBeenCalledWith(...)  // 内部ヘルパーへの呼び出しを assert
```
```python
mock_internal.assert_not_called()
assert mock_internal.call_count == 0
```
内部の呼び出しトレースは観測不可。実装を別アプローチに書き換えると壊れる。

**書くべきもの**: その呼び出しが生んだ外部から見える結果（戻り値・永続化・例外）。

### 違反: 組み立てた文字列の中身を assert
```js
expect(buildQuery(filter)).toBe("SELECT * FROM users WHERE id = 42")
expect(logged).toContain("user_id=42 action=login")
```
クエリ文字列・ログ文字列の組み立て方は内部詳細。同じデータを返す別のクエリに書き換えると壊れる。

**書くべきもの**: クエリが返すデータ（DB 状態 → 公開 API レスポンス）、ログが影響する外部接点（監視ルール）の動き。

### 違反: 内部生成 ID / 連番の値を assert
```js
expect(record.internal_id).toBe(1)
expect(generated.uuid).toMatch(/^\d{6}$/)  // 実装の連番ロジックを写している
```
内部生成値は実装が変わると変わる。

**書くべきもの**: その ID が外部に返るなら、契約として固定されている形のポジティブな assert（例: 「UUID v4 形式」と外部スキーマで宣言済みなら、その形と一致することを assert）。実装が選んだ内部ロジック（連番・特定パターン）は固定しない。

### 違反: private 状態への到達を確認
```js
expect((obj as any)._cache.size).toBe(3)
expect(service._state).toBe("ready")
```
private は観測不可。別実装に書き換えると消える。

**書くべきもの**: その状態がもたらす外部から見える結果（キャッシュヒット時の応答時間・レスポンス内容）。

### 違反: コードのトポロジーを確認
```js
expect(typeof module.exports.helper).toBe("function")  // 存在確認だけ
```
公開 API の一部として export されている場合を除き、ただの存在チェックは振る舞いを検証していない。

## 「不在を assert したい」と感じたときは、たいていテスト不要

不在を assert したくなった瞬間、それは差分テストの可能性が高い。書く前に「自分はなぜこれをテストしたいのか」を問い直す。判別の問い:

> **「現在こう動く」を固定したいのか、「以前と違って今はこう」を固定したいのか？**

後者ならテスト不要。git diff・スキーマファイル・コードレビュー・PR 履歴で既に表現されている事実を、テストで重ねても情報量は増えない。「不在を別の形でポジティブに書き換える」ことを考えても、動機が「あったものが無くなったことを縛りたい」のままだと結局差分テストにしかならない。

### 多くの場合: テスト不要

| 縛りたい意図 | テスト不要な理由 |
|---|---|
| 削除されたメソッド／クラス／プロパティの不在 | `git diff` が正本。テストで重ねても情報は増えず、誰かが再追加した時にしか壊れない |
| 404 化されたエンドポイントの「存在しない」確認 | HTTP の default 挙動（未定義パスは 404）と区別できない。ハンドラ削除と「元から無い」を見分けられない |
| drop された DB カラムの「無い」確認 | スキーマファイル（DDL / migration）が正本。テストで重ねても冗長 |
| 削除されたログ出力の「出ていない」確認 | 何が出るかを別途固定すれば十分。「これは出ない」だけを縛る必要はない |
| 削除メソッドの復活防止 | 復活して困るのは外部から観測される問題が起こるから。それを直接テストするのが筋（不在の確認ではなく） |
| migration 後のスキーマを「期待値と完全一致」で assert | スキーマファイルそのものと冗長。「以前は X があった→今は無い」という差分を固定しているだけ |

### 例外: 新契約をポジティブに固定したい場合

「不在を縛りたい」ではなく「**現在こういう振る舞いを保証したい**」が動機なら、振る舞いベースで書ける。判定基準を両方満たし、不在 assert より漏れにも強い（別名で漏らしたケースも検知できる）:

- **レスポンスのフィールド集合を allowlist で完全一致**: `expect(response).toEqual({id, email, name})`。「機密情報を含まない」より「現在の契約はこれ」というポジティブな固定。将来 `pw_hash` や `secret` で漏らしても allowlist 不一致で検知
- **ログの構造化フィールド集合を allowlist で完全一致**: 「PII を出さない」より「ログには `{user_id, action, status}` のみ含まれる」というポジティブな固定
- **404 化したエンドポイントを使う側の振る舞い**: サーバー側で 404 を assert するのではなく、**呼び出し側コードで「`/legacy/endpoint` への呼び出し時にエラーハンドリングが動く」** をテスト。これは呼び出し側にとっての観測可能な振る舞いであって、削除ではなく新しい使い方の契約
- **削除されたメソッドが担っていた要件の新しい契約**: 「削除した」ことを縛るのではなく、「新しい設計でこの要件はこう実現される」をテスト。動機を差分から「現在の契約」に切り替える

### 判別の補助線

| パターン | 振る舞いベース | 差分テスト |
|---|---|---|
| 「現在の状態が X である」 | ◯（X を固定） | — |
| 「過去は Y だったが今は X」 | — | ✗（差分を固定） |
| 「X 以外を含まない（allowlist 完全一致）」 | ◯（集合を固定） | — |
| 「Y を含まない（denylist 単項）」 | △（Y 以外の漏れを検知不能） | ✗（特定差分を縛る） |
| 「呼び出し側が X を観測する」 | ◯（呼び出し側の契約） | — |
| 「コード／スキーマに X が存在しない」 | — | ✗（git diff / DDL と冗長） |

### それでも翻訳できないと感じたとき

判別の問いを当てても「現在こう動く」と「以前と違って今はこう」の境目が曖昧な場合は、**書かない側に倒す**。書かないことで失うものは git diff・PR 履歴・スキーマファイルが既に担保しているはず。書くことで得られるはずの情報がそれらと冗長なら、メンテナンス負債だけが残る。

## レビュアー（agent）が指摘を出すときのチェック

コードレビュアー（code-reviewer-*）など、テスト追加を指摘する側は、**自分の推奨対応が振る舞いベースになっているか** を出力前にセルフチェックする。

推奨対応欄に書く言葉が次のいずれかになっていたら、判定基準を当て直す:

- 「`X` が呼ばれない／削除されていることを確認するテストを足せ」
- 「`X.foo()` が `Y.bar()` を呼ぶことを spy で確認」
- 「組み立てたクエリ／ログの文字列が一致することを assert」
- 「private プロパティ `_state` の値を確認」

→ どれも基準 1 / 基準 2 のどちらかに引っかかる。観測可能な振る舞いに翻訳できるなら推奨対応を書き直す。翻訳不能なら、その指摘を出さない（confidence を下げて落とす）。

## 統合（integrator）が指摘をふるい分けるときの基準

複数レビュアーの指摘を集約する側（code-review-integrator）は、テスト系指摘について判定基準を当てて、振る舞いに翻訳できないものは下流に流さない。詳細な工程は integrator の指示書側で定義する。

## よくある誤適用

- **「単にカバレッジを上げたい」**: カバレッジは観測指標であって目的ではない。基準を当てずに「テストが無いから足す」は、差分追従テストを量産する近道
- **「リファクタを安全にしたいから内部実装をテストで固定したい」**: 振る舞いベースのテストが充実しているなら、リファクタは振る舞いベースのテストで安全になる。内部実装を固定するテストはリファクタを阻害する
- **「将来の誰かが間違って復活させないように削除されたコードをテストで縛りたい」**: その復活を防ぎたいのは、復活すると **外部から観測される問題** が起こるからのはず。それを直接テストする
