## テストシナリオ一覧

振る舞いの仕様から導出される具体的なテストケースをテーブルで列挙する。テスト実装者はこのテーブルを参照してテストを作成する。Phase構造ではなくテストファイル単位でまとめ、各テストケースには対応する振る舞いへの参照タグ（例: `[1.1]`）を付ける。

### 手順0: 既存テストのカバレッジ地図（シナリオを列挙する前に必須）

**シナリオ表を書く前に、今回触る振る舞いの周辺にある既存テストを実際に開いて読み、何が既に担保されているかを地図にする。** これを飛ばすと「既にカバーされている観点」を新規シナリオとして重複追加してしまう（テスト水増しの最大の原因）。

- **調査範囲は「触る領域」に絞る**（全テストを調べる必要はない）。今回の各 Phase で定義した振る舞いが影響する範囲 — 同じモジュール・同じ関数・同じエンドポイント・同じバリデータ・隣接する正常/異常経路 — の既存テストに限定する。
- 読んだ結果を次の地図にする。これが後段の「どの観点は既に守られているから新規シナリオ不要か」の根拠になる。

| 既存テスト | 担保している観点 |
|---|---|
| `path/to/test.py::test_name` | （このテストが現在保証している振る舞い） |

### テストシナリオ選定の原則（重要）

テストシナリオは「カバレッジを上げる」ためでなく「振る舞いの意図を守る」ために選ぶ。テーブルに列挙する前に、各候補テストを以下の問いで必ずフィルタすること。

1. **意図テスト**: そのテストが落ちたとき、どの振る舞い仕様が壊れたかを一言で言えるか？ 言えなければ削る。実装詳細・内部状態の確認はテストにしない
2. **代表テスト**: enum値・分岐パターンの機械的な網羅は避ける。「STOP以外で警告が出る」を検証したいなら enum 種別ごとにテストを増やさず、代表1件 + 境界値で十分。**同じ関数・同じバリデータ・同じヘルパを通る複数経路（例: create と update が同一バリデータを呼ぶ、複数の early-return 分岐が同一エラーヘルパを使う）は、片方を代表として1件にする**
3. **既存担保テスト**: 上のカバレッジ地図に載っている既存テストがその観点を既に担保しているなら、新規シナリオにしない。「Aが動けばBも動く」が対称性で導かれるなら片方のみ
4. **独立性テスト**: 1つのテストが複数の振る舞いを混ぜて検証していないか？ 混ざっていれば、本当に検証したい1つに絞るか、テストを分割する
5. **回帰確認は新規に足さない**: **今回変えない既存の振る舞いについて「従来どおり動く」ことを確認する回帰テストは新規追加しない。** その回帰はカバレッジ地図に載っている既存テストが検知する。新規に書くのは「今回新しく生まれる振る舞い」と「今回変わる振る舞い」だけ。「自分が触る周辺だから念のため守っておく」は重複の温床

**全振る舞いに担保元を1つ持たせる。** 計画の各振る舞いタグ `[x.y]` は、必ず「新規シナリオ行」または「カバレッジ地図の既存テスト」のどちらかで担保された状態にする。既存テストで担保されるものは新規シナリオを作らない。これを徹底すると「既にカバーされている観点をうっかり新規追加する」のを機械的に防げる。

人間のレビューで「テスト数が多すぎる」「振る舞いより網羅を見ている」と指摘された場合は、このセクションに立ち返って選定をやり直すこと。

**重要: 各行の先頭に通し番号（`#`）を付ける。** レビュー時に「3番のテストが〜」のように特定のシナリオを参照しやすくするため。番号はテーブル全体で1から連番で振る（テストファイルごとのリセットはしない）。

**重要: 区分の表記ルール。** テストファイル列には以下の3種類の区分を必ず付ける。
- `（既存修正）`: 既存テストの内容を変更する場合。**どのテストを修正するかを `path/to/file.py::test_name` 形式で必ず明記する**（テスト実装者が対象を一意に特定できるようにするため）
- `（追記）`: 既存テストファイルに新しいテストケースを追加する場合
- `（新規）`: 新規テストファイルを作成する場合

| # | テストファイル | テスト内容 | テストしたい振る舞い |
|---|---|---|---|
| 1 | `path/to/existing_test.py::test_existing_name`（既存修正） | 既存テストをどう変更するか | [1.1] 対応する振る舞いの要約 |
| 2 | `path/to/existing_test.py`（追記） | input1="valid", input2=10 → 期待される結果 | [1.2] 対応する振る舞いの要約 |
| 3 | `path/to/existing_test.py`（追記） | input1="" → ValidationError | [1.2] 対応する振る舞いの要約 |
| 4 | `path/to/new_test.py`（新規） | input=MAX_LIMIT → 正常処理 | [2.1] 対応する振る舞いの要約 |
| 5 | `path/to/new_test.py`（新規） | input=MAX_LIMIT+1 → ValidationError | [2.1] 対応する振る舞いの要約 |

### 除外したシナリオと担保元（必須）

検討したが新規追加しなかったシナリオを、担保元とともに記録する。**「何を書いたか」だけでなく「何を意図的に書かなかったか・なぜか」を残すことで、人間とレビュアーが選定の妥当性を検証できる。** カバレッジ地図と合わせて、これが「網羅ではなく意図で絞った」ことの証拠になる。検討の結果ゼロ件除外なら「なし」と明記する。

- （検討したシナリオ）→ 担保元: （カバレッジ地図の既存テスト名 / 代表テストに集約 / 対称性で別シナリオが担保 / 実装詳細寄りのため見送り 等）
