## 実装の進め方

### 1. 計画の理解
- 要件と実装計画を理解
- 実装方針を確認
- 変更予定ファイルを把握
- 完了条件を確認

### 2. 実装
- 計画に忠実に実装
- 既存のコーディングスタイルに従う
- 適切なエラーハンドリング

### 3. テスト
- **判定基準を当てる**: テストを書く前に `workflow-components/test/test-discipline.md` を読み、判定基準（**置換不変性** / **観測可能性**）を両方満たすテストだけを書く
- **既存テストへの追記を第一選択**: 新規ファイルを作る前に、既存テストファイル／クラスへの追記で済むかを必ず先に検討
- **既存テストの実行**: リグレッションがないことを確認
- **エッジケース**: 観測可能な異常系（送出される例外の種別・公開 API のエラーレスポンス等）もテスト
- **人間レビュー指示の翻訳**: 人間から「`X` の不在を assert して」「`Y` が呼ばれないことを確認して」のような指示を受けた場合、fixer の「指摘の翻訳工程（必須）」に従って観測可能な振る舞いに翻訳するか、対応見送りを人間に返す
- 既存のテストスタイルに従う

### 4. 品質チェック
- lint を実行してエラーがないこと
- 型チェック（該当する場合）
- 全テストがパスすること
